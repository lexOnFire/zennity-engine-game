# ADR-002 — Arquitetura Baseada em Módulos/Plugins

**Status:** Aceito • **Data:** 2026-07-01

---

## Contexto

Engines monolíticas embarcam toda funcionalidade no core: render, física, áudio,
networking. Isso cria acoplamento forte, dificulta testes e impede que a
comunidade adicione extensões sem modificar o código principal.

## Decisão

A Zennity é uma **plataforma de módulos**, não um conjunto de funcionalidades embutidas.

- O `engine.core` é pequeno e estável: apenas Application, Scene, GameObject,
  Component, System, Time, EventBus, Logger, SceneManager, Engine.
- Render 2D, Render 3D, Física, Áudio, UI, Networking, IA existem como
  **módulos independentes** que o usuário habilita conforme necessidade.
- Cada módulo registra seus sistemas na `Application` via `SystemRegistry`.
- O Core nunca importa de módulos. Módulos importam do Core.

```python
# Habilitar apenas o que você precisa
app = Application(800, 600, "Meu Jogo")
app.systems.add(PhysicsSystem())
app.systems.add(Renderer2D())
app.run(MenuScene())
```

## Consequências

**Positivas:**
- Zero acoplamento entre sistemas
- Testável: core roda sem janela, física, ou render
- Extensível: comunidade pode criar módulos sem tocar no core
- Builds menores: inclui apenas o que usa

**Negativas:**
- Configuração inicial mais verbosa para jogos simples
- Exige que módulos sigam a API de `System` e `SystemRegistry`

## Alternativas Consideradas

- Engine monolítica com tudo embutido (rejeitado: dificulta manutenção e extensão)
- IoC container externo como `dependency_injector` (rejeitado: dependência pesada para algo simples)
